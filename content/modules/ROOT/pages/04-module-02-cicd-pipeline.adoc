== Part 2 — Pipeline catches code smells

=== Know

_The pipeline's SAST/Sonarqube stage fails because the developer's new claims statistics endpoint contains code smells: `System.out.println` instead of proper logging and an empty catch block that silently swallows exceptions. Sonarqube's quality gate enforces coding standards, and these violations prevent the code from moving forward. This is the platform's guardrail in action: the CI pipeline enforces code quality standards that the developer's inner loop does not. The developer fixes the issues in DevSpaces, pushes again, and the pipeline passes._

**Business challenge:**

* Code smells and lack of automated code quality enforcement accumulates technical debt, making the codebase harder to maintain over time
* Improper logging (`System.out.println`) makes production debugging expensive and time-consuming
* Swallowed exceptions hide failures, causing silent data corruption or incorrect behavior
* Manual code review alone cannot catch all quality issues consistently across teams

**Value proposition:**

Automated quality gates in the pipeline catch code smells that manual processes miss. Sonarqube enforces organizational coding standards — proper logging frameworks, meaningful exception handling, code complexity limits — on every commit. When the pipeline fails, the developer gets specific, actionable feedback and can fix the issues immediately in DevSpaces. The platform enforces quality without becoming a bottleneck.

package com.parasol;

import java.util.HashMap;
import java.util.Map;

import io.quarkus.logging.Log;
import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;
import jakarta.ws.rs.Produces;
import jakarta.ws.rs.core.MediaType;

@Path("/api/claims/stats")
@Produces(MediaType.APPLICATION_JSON)
public class ClaimsStatsResource {

	private final ClaimsResource claimsResource;

	public ClaimsStatsResource(ClaimsResource claimsResource) {
			this.claimsResource = claimsResource;
	}

	@GET
	public Map<String, Object> getStats() {
			Map<String, Object> stats = new HashMap<>();

			Log.info("Fetching claim statistics");

			var claims = claimsResource.getAllClaims();
			stats.put("total", claims.size());

			Map<String, Long> byCategory = new HashMap<>();
			for (var claim : claims) {
					byCategory.merge(claim.getCategory(), 1L, Long::sum);
			}
			stats.put("by_category", byCategory);

			Map<String, Long> byStatus = new HashMap<>();
			for (var claim : claims) {
					byStatus.merge(claim.getStatus(), 1L, Long::sum);
			}
			stats.put("by_status", byStatus);

			return stats;
	}
}

* Manual deployments are error-prone and difficult to audit
* Configuration drift between environments causes "works in staging, fails in production" issues

== Part 4 — OpenShift Builds: platform-governed image building

=== Know

_In Part 1, the pipeline built and containerized the Parasol application automatically using OpenShift Builds with Cloud Native `Buildpacks` -- without a `Containerfile`. This part explores that build process in more detail: what happened under the hood, how the platform engineer governs the build process, and how the build metrics provide visibility into CI performance._

**Business challenge:**

* Container image builds are often ungoverned -- developers write their own `Dockerfiles` with inconsistent base images, configurations, and security practices
* Platform teams have no visibility into build performance or resource consumption
* `Containerfile` maintenance becomes a burden as frameworks and base images evolve
* No standardized, auditable way to control _how_ applications are built across teams

**Value proposition:**

OpenShift Builds (based on the Shipwright project) separates the concerns of _what_ to build from _how_ to build. Platform engineers define approved build strategies (`ClusterBuildStrategy` resources) that govern base images, security contexts, and resource limits. Developers just point at their source code -- the platform handles the rest. With Cloud Native `Buildpacks`, developers do not even need a `Containerfile`: the `buildpack` auto-detects the application type and builds the image using best practices for that framework.

**Business value callout:**

"This is the platform doing the undifferentiated heavy lifting. At Parasol today, every team maintains their own `Dockerfiles`, often with different base images, different build patterns, and different levels of security hygiene. When a base image CVE is announced, someone has to track down every `Dockerfile` across every team and update them. With OpenShift Builds, the platform team manages the build strategies centrally. When a base image needs updating, they update the strategy in one place and every team's builds use the new image on their next run. Developers never think about `Dockerfiles` -- they write code, and the platform builds it safely."

**If asked:**

Q: "What if we need a custom build strategy for a specific use case?"::
A: "Platform engineers can create namespace-scoped `BuildStrategy` resources for specific teams or use cases. These follow the same model but are scoped to a single namespace. For example, a team that needs GPU-accelerated builds could have a custom strategy with different resource limits."

Q: "Can we still use `Dockerfiles` if we want to?"::
A: "Yes. The `buildah` ClusterBuildStrategy supports traditional `Dockerfile`/`Containerfile`-based builds. Teams can migrate to `buildpacks` at their own pace. The governance model is the same either way: the platform controls the strategy, the developer references it."

Q: "What about build caching?"::
A: "Cloud Native `Buildpacks` support layer caching across builds. Dependencies that have not changed are reused from previous builds, significantly reducing subsequent build times. The `buildpacks` strategy supports both PVC-based and image-based caching."
